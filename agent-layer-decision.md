# Agent Layer Decision

Срез: 2026-03-19

## Решение

На текущем этапе **не добавлять `sgr-agent-core`** в целевую архитектуру.

Основной инструмент:

- `OmniRoute`

Отложенное решение:

- `sgr-agent-core` рассматривать только как возможный **второй этап**

## Почему

Текущий пользовательский сценарий:

- кодинг
- мультипровайдерная агентская разработка
- удобное переключение между моделями и провайдерами
- практическая польза здесь и сейчас

Эту задачу уже закрывает `OmniRoute`.

`sgr-agent-core` добавляет отдельный agent framework слой, а значит:

- ещё один сервис
- ещё одну точку отказа
- более сложный дебаг
- больше архитектурной нагрузки

При этом на текущем этапе нет отдельной цели:

- строить свой research-agent backend
- поднимать отдельный агентный API для команды
- автоматизировать многошаговые research workflow как отдельный продукт

## Практический вывод

Сейчас нужно:

1. поднять и стабилизировать `OmniRoute`
2. использовать его как основной multi-provider gateway
3. не усложнять архитектуру вторым агентным слоем без явной необходимости

## Когда вернуться к `sgr-agent-core`

Возвращаться к этой идее имеет смысл только если появится отдельная задача:

- собственный research-agent сервис
- кастомные многошаговые агенты
- повторяемые deep research workflow
- внутренний агентный API для команды

До этого момента `sgr-agent-core` считать не обязательным и отложенным.

## Финальная формулировка

Для текущего сценария:

- `OmniRoute` — да
- `sgr-agent-core` — пока нет

Это осознанное упрощение архитектуры, а не пропуск важного компонента.
