# One Page Summary

Срез: 2026-03-20

## Запрос

Нужно выбрать open-source решение, похожее на `OmniRoute`, для работы с несколькими LLM-провайдерами и гибкого переключения между ними.

Позже запрос уточнился:

- основной сценарий это кодинг
- нужен инструмент для мультипровайдерной агентской разработки
- есть положительный рабочий референс по `OmniRoute` от коллеги

Позже в исследование был добавлен еще один очень релевантный кандидат:

- `CLIProxyAPI`

## Короткий вывод

Если смотреть абстрактно и универсально, самый сильный общий кандидат: `LiteLLM`.

Если смотреть именно под ваш сценарий, финальная пара выбора: `CLIProxyAPI` и `OmniRoute`.

По строгому fit под AI coding, CLI/OAuth и мультипровайдерную агентскую разработку вперед выходит `CLIProxyAPI`.

## Почему не просто LiteLLM

`LiteLLM` оказался лучшим по dry run, зрелости и предсказуемости как универсальный gateway.

Но ваш сценарий уже не про “универсальный gateway вообще”.

Он про:

- coding workflow
- агенты
- CLI/developer tooling
- быстрое ручное переключение между провайдерами

В этом контексте лучше попадают `CLIProxyAPI` и `OmniRoute`, а `CLIProxyAPI` выходит вперед по прямому fit под CLI/OAuth coding workflow.

## Практический выбор

Основной выбор для экстремального вайбкодинга:

- `CLIProxyAPI`

Альтернативный выбор, если нужен более цельный встроенный dashboard/control plane:

- `OmniRoute`

Инфраструктурный план B:

- `LiteLLM`

## Что это значит на практике

Если вы хотите максимум прямого попадания в CLI/OAuth coding workflow, начинайте с `CLIProxyAPI`.

Если вам важнее более цельный self-hosted UI и есть сильный референс от коллеги, начинайте с `OmniRoute`.

Если позже захотите более нейтральный инфраструктурный слой, запасной вариант это `LiteLLM`.

## Финальная рекомендация

Под ваш реальный use case я советую:

1. для pure coding/CLI use case брать `CLIProxyAPI`
2. если нужен UI-first developer gateway брать `OmniRoute`
3. держать `LiteLLM` как инфраструктурный план B
