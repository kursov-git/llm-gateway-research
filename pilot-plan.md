# Pilot Plan

Срез: 2026-03-18

## Цель пилота

Не выбирать gateway по README и звездам, а проверить 2-3 кандидата на ваших реальных сценариях:

- работа с несколькими LLM-провайдерами
- гибкое переключение
- простота настройки и сопровождения
- наблюдаемость и дебаг

## Кого проверять

Основной shortlist для пилота:

1. `LiteLLM`
2. `OmniRoute`
3. `uni-api`

Опционально вместо `uni-api`:

- `Portkey Gateway`, если важны policy/guardrails/governance

## Формат пилота

Для каждого кандидата поднимать отдельное изолированное окружение и прогонять одинаковый набор сценариев.

Оценка по 5-балльной шкале:

- установка
- настройка
- routing/fallback
- наблюдаемость
- удобство эксплуатации

## Набор сценариев

### Сценарий 1. Базовый старт

Проверить:

- насколько быстро поднимается первый рабочий инстанс
- сколько ручных шагов нужно до первого успешного запроса
- есть ли понятный health/status сигнал

Что фиксировать:

- время до первого успешного запроса
- число обязательных шагов
- число мест, где можно ошибиться

### Сценарий 2. Подключение нескольких провайдеров

Проверить:

- добавить минимум 3 разных провайдера
- например: `OpenAI`, `Anthropic`, `OpenRouter` или другой ваш реальный набор
- убедиться, что все доступны через единый API endpoint

Что фиксировать:

- способ подключения
- где хранится конфигурация
- можно ли переиспользовать конфиг между окружениями

### Сценарий 3. Fallback при ошибке

Проверить:

- основной провайдер возвращает ошибку
- gateway корректно переключается на резервный
- ошибка видна в логах и понятна в диагностике

Что фиксировать:

- поддерживается ли fallback из коробки
- насколько явно управляется цепочка fallback
- нужно ли писать кастомную логику

### Сценарий 4. Rate limit и временная деградация

Проверить:

- как система ведет себя при `429`
- есть ли cooldown, retry, provider/key rotation
- можно ли избежать повторного удара в “сломанный” upstream

Что фиксировать:

- наличие retry
- наличие circuit breaker или эквивалента
- читаемость operational signals

### Сценарий 5. Добавление нового провайдера

Проверить:

- сколько времени занимает подключение нового провайдера
- нужны ли нестандартные адаптации
- насколько удобно это делать повторно

Что фиксировать:

- изменение через UI, YAML, env или код
- требуется ли restart
- есть ли hot reload

### Сценарий 6. Логи и диагностика

Проверить:

- видно ли, какой upstream был выбран
- видно ли, почему произошел fallback
- можно ли быстро найти причину ошибки запроса

Что фиксировать:

- наличие dashboard/log UI
- качество request logs
- пригодность для инцидентного разбора

### Сценарий 7. Безопасность и эксплуатация

Проверить:

- как задаются ключи доступа
- есть ли API keys, auth, basic rate limiting
- не остаются ли небезопасные defaults после первого старта

Что фиксировать:

- что нужно менять сразу после установки
- насколько production-safe дефолтная конфигурация

## Критерии победы

Решение проходит пилот, если:

- поднимается меньше чем за 1 час без экзотической ручной доработки
- стабильно работает минимум с 3 нужными провайдерами
- fallback и retry не требуют ручного кода вокруг него
- диагностика инцидентов занимает минуты, а не часы
- конфигурация воспроизводима между dev и prod

## Практический порядок выполнения

1. Поднять `LiteLLM`.
2. Поднять `OmniRoute`.
3. Поднять `uni-api`.
4. Прогнать одинаковый сценарный чек-лист.
5. Свести результат в weighted score.
6. Оставить 1 основной и 1 резервный кандидат.
