# Research Brief

Срез: 2026-03-18

## Исходный запрос пользователя

Изучить репозиторий `OmniRoute` и на его фоне подобрать похожее решение для работы с разными LLM-провайдерами и возможностью гибкого переключения между ними.

Нужно:

- изучить сильные стороны `OmniRoute`
- изучить слабые стороны `OmniRoute`
- найти похожие open-source репозитории
- подготовить сравнение
- помочь выбрать наиболее подходящий вариант

## Что именно имелось в виду под задачей

Задача не про абстрактную “AI platform”.

Задача про практический слой для:

- работы с несколькими LLM-провайдерами
- единой точки входа для запросов
- переключения между провайдерами и моделями
- fallback / routing / failover
- удобства именно в повседневной разработке

## Уточнения, полученные в ходе диалога

Позже пользователь уточнил более важный прикладной контекст:

- основной сценарий это **кодинг**
- интересует именно **мультипровайдерная агентская разработка**
- есть **реальный референс от коллеги**, который уже использует `OmniRoute` как рабочий инструмент

Это изменило приоритеты выбора:

- более важным стал не “самый зрелый универсальный gateway вообще”
- а “лучший инструмент под developer-first / agent-first workflow”

## Практическая формулировка запроса после уточнений

Если переформулировать задачу так, как она фактически сложилась к концу исследования:

> Нужно выбрать open-source gateway/router для повседневной агентской разработки, где удобно работать с несколькими LLM-провайдерами, быстро переключаться между ними, использовать fallback и не тратить время на ручную переконфигурацию инструментов.

## Неявные требования, выявленные по ходу обсуждения

На основании диалога и реакции пользователя важны следующие свойства:

- решение должно быть **практичным**, а не исследовательским
- должно быть пригодно для **реального coding workflow**
- желательно, чтобы оно уже имело **живые подтверждённые use cases**
- важна не только “поддержка провайдеров”, но и **удобство эксплуатации**
- решение должно быть релевантно для **агентных CLI / developer tools**

## Что в итоге стало ключевым критерием выбора

Ключевой критерий сместился к следующему:

> Насколько решение подходит именно для developer-first multi-provider agentic coding workflow, а не просто для generic LLM gateway use case.

## Итоговая рекомендация по этому запросу

Основной выбор под этот запрос:

- `OmniRoute`

Резервный вариант:

- `LiteLLM`

Причина:

- `LiteLLM` выглядит сильнее как универсальный gateway
- но `OmniRoute` лучше совпадает с уточнённым сценарием пользователя: coding, CLI, агенты, мультипровайдерная ежедневная разработка

## Что уже было сделано в рамках этого исследования

Выполнено:

- обзор `OmniRoute`
- поиск и shortlist похожих open-source решений
- сравнительная матрица
- decision memo
- dry run по `OmniRoute`, `LiteLLM`, `uni-api`
- фиксация результатов в репозитории исследования

## Что нужно сделать, если исследование будут повторять новые агенты

Если в будущем это исследование будет выполняться заново, важно не потерять контекст пользователя.

Новые агенты должны стартовать с таких вводных:

1. Это не общий market scan по AI gateways.
2. Это выбор инструмента под **agentic coding workflow**.
3. Пользователь склоняется к `OmniRoute`, потому что:
   - хочет кодить
   - интересуется мультипровайдерной агентской разработкой
   - имеет положительный внешний референс от коллеги
4. Исследование должно сравнивать:
   - широту провайдеров
   - гибкость routing/fallback
   - удобство настройки
   - пригодность к ежедневной разработке
   - зрелость и предсказуемость эксплуатации
5. В финальном выводе нужно разделять:
   - “универсально лучший gateway”
   - “лучший инструмент именно под coding/agents”

## Рекомендуемый формат для повторного исследования

Если делать следующий исследовательский проход:

1. Актуализировать shortlist.
2. Проверить, не появился ли новый сильный developer-first кандидат.
3. Перепроверить `OmniRoute` и `LiteLLM` на текущем состоянии экосистемы.
4. Провести короткий dry run.
5. По возможности провести real pilot с реальными провайдерами пользователя.

## Вопрос, который нужно задать в начале следующего прохода

Чтобы новый агент не ушёл в слишком широкий ресёрч, ему стоит сразу уточнить:

> Вам нужен лучший универсальный multi-provider gateway или лучший инструмент именно для coding/agent workflow?

Для этого пользователя на момент данного исследования ответ уже известен:

> Нужен инструмент именно для coding и мультипровайдерной агентской разработки.
