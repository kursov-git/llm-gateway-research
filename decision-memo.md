# Decision Memo

Срез: 2026-03-18

## Вопрос

Какое open-source решение выбирать для работы с несколькими LLM-провайдерами и гибкого переключения между ними, если нужен practical self-hosted вариант без лишнего исследовательского шума.

## Короткое решение

Базовый выбор: `LiteLLM`.

Альтернативный выбор для developer-first и coding-agent сценариев: `OmniRoute`.

Легкий config-as-code выбор: `uni-api`.

## Решение по сценариям

### Сценарий 1. Нужен один основной gateway для команды и приложений

Выбирать `LiteLLM`.

Почему:

- Это самый зрелый и универсальный OSS слой из найденных.
- У него сильная поддержка множества провайдеров.
- Он хорошо совпадает с задачей “единый gateway + routing + load balancing + spend control”.
- Конфигурация воспроизводимая и понятна для инфраструктурного сопровождения.

Что принять заранее:

- Это Python-first стек.
- Для полноценного production-профиля он может потребовать больше инфраструктуры, чем более легкие решения.

### Сценарий 2. Основной use case это разработчики, IDE, CLI и coding agents

Выбирать `OmniRoute`.

Почему:

- У него выраженный уклон в developer tooling.
- Он хорошо закрывает сценарии `Codex`, `Claude Code`, `Gemini CLI`, похожие CLI и fallback между ними.
- UI и operational surface удобны именно для живого ручного использования.

Что принять заранее:

- Это не самый “консервативный” gateway.
- Для production и особенно публичной экспозиции нужен отдельный security review и hardening.
- Поддержка некоторых сценариев будет более чувствительна к изменениям внешних клиентов и провайдеров.

### Сценарий 3. Нужен легкий router без тяжелой панели и без платформенности

Выбирать `uni-api`.

Почему:

- Он хорошо подходит под config-as-code модель.
- `api.yaml` и `CONFIG_URL` позволяют прозрачно управлять конфигурацией.
- Легко держать в git, ревьюить и воспроизводить между окружениями.

Что принять заранее:

- Меньше UI и меньше control-plane возможностей.
- Если команде нужен удобный визуальный ops-интерфейс, этого будет не хватать.

### Сценарий 4. Нужны policy routing, guardrails и governance

Выбирать `Portkey Gateway`.

Почему:

- Он сильнее остальных выглядит именно как policy/gateway слой.
- Подходит, если routing это часть более широкой задачи reliability, control и guardrails.

Что принять заранее:

- Это менее “минималистичный self-host router”, чем `uni-api`.
- Восприятие и ecosystem fit сильнее завязаны на продуктовую экосистему Portkey.

## Что не брать первым выбором

### One API

Не как основной кандидат для вашей задачи.

Почему:

- Это полезный self-hosted API hub.
- Но он ближе к channel management и key distribution, чем к лучшему варианту для гибкого routing layer.

### Helicone AI Gateway

Не как первый кандидат без дополнительного пилота.

Почему:

- Выглядит интересно по performance и observability.
- Но по совокупности зрелости и предсказуемости выбора на сегодня уступает `LiteLLM` как default choice.

### RouteLLM

Не как замена gateway.

Почему:

- Это сильный intelligent router framework.
- Но это не полноценный multi-provider ops gateway.

## Практическая рекомендация

Если нужно быстро сузить выбор до трех:

1. `LiteLLM` как основной инфраструктурный кандидат.
2. `OmniRoute` как developer-first кандидат.
3. `uni-api` как легкий YAML-first кандидат.

Если нужно сузить до одного:

Выбирать `LiteLLM`, если нет явной причины оптимизироваться под coding-agent/CLI workflow.

Выбирать `OmniRoute`, если основная ценность именно в ручном управлении fallback между developer-oriented провайдерами и аккаунтами.

## Следующий шаг

Логичный следующий шаг перед окончательным выбором:

1. Проверить 2 кандидата на вашем реальном наборе провайдеров.
2. Зафиксировать желаемую модель конфигурации: UI-first или config-as-code.
3. Прогнать пилот по 4 критериям:
   - добавление нового провайдера
   - fallback при ошибке и rate limit
   - наблюдаемость и дебаг запросов
   - простота обновления и сопровождения
